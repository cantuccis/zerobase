# Settings API for Server Configuration

**Date:** 2026-03-21
**Task ID:** he0rfgix7l2t1li
**Phase:** 5

## Summary

Implemented admin endpoints for managing server-wide settings (SMTP, S3, auth, application metadata, backup configuration). Settings are stored as key-value JSON entries in the existing `_settings` table, accessible only by superusers.

## Architecture

### Settings Categories (well-known keys)
- **meta** — Application name, URL, sender name/address
- **smtp** — SMTP email configuration (host, port, TLS, credentials)
- **s3** — S3-compatible storage settings (bucket, region, endpoint, credentials)
- **backups** — Backup scheduling and S3 configuration
- **auth** — Token duration, password requirements, auth method toggles

### Layers
1. **SettingsRepository trait** (zerobase-core) — persistence contract with get/set/delete/list operations
2. **SettingsService** (zerobase-core) — domain logic: validation, deep-merge updates, defaults for missing keys
3. **Settings handlers** (zerobase-api) — HTTP handlers for GET/PATCH/DELETE
4. **Settings routes** (zerobase-api) — Router with superuser middleware protection

### Endpoints
- `GET /api/settings` — Get all settings (returns defaults for missing keys)
- `PATCH /api/settings` — Partial update (deep-merge per category)
- `GET /api/settings/{key}` — Get single setting category
- `DELETE /api/settings/{key}` — Reset setting to default

## Key Design Decisions
- **Deep merge**: PATCH requests merge at the object level, preserving unmentioned fields
- **Defaults**: Missing settings return sensible defaults (not errors)
- **Validation**: SMTP requires host when enabled; S3 requires bucket+region when enabled
- **Unknown keys rejected**: Only well-known setting keys are accepted
